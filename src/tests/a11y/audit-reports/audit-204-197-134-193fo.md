## Issue #197 — BulkActionBar: confirm-delete focuses Cancel instead of destructive button

**File:** `src/ui/Sidebar.tsx`, `BulkActionBar` component (approx. lines 777–888)

### Finding 1: Focus routed to Cancel on confirm-delete open — PASS

The implementation uses the correct pattern:

```tsx
const confirmCancelRef = useRef<HTMLButtonElement>(null);
const deleteSelectedRef = useRef<HTMLButtonElement>(null);

useEffect(() => {
  if (barState === 'confirm-delete') {
    confirmCancelRef.current?.focus();
  }
}, [barState]);
```

When `barState` transitions to `'confirm-delete'`, the `useEffect` fires and programmatically focuses the Cancel button. This is the correct approach — `useEffect` with `barState` as dependency ensures the DOM has rendered the confirmation UI before focus is applied. The Cancel button is the safe default for a destructive action. WCAG 2.4.3 (Focus Order) satisfied.

### Finding 2: Focus returns to logical target on dismiss — PASS

Both the cancel path and the confirm path return focus to `deleteSelectedRef`:

```tsx
const handleCancelConfirm = useCallback(() => {
  setBarState('idle');
  setTimeout(() => deleteSelectedRef.current?.focus(), 0);
}, []);

const handleBulkDeleteConfirm = useCallback(() => {
  onBulkDelete();
  setBarState('idle');
  setTimeout(() => deleteSelectedRef.current?.focus(), 0);
}, [onBulkDelete]);
```

The `setTimeout(..., 0)` defers the focus call to the next tick, after `barState` returns to `'idle'` and React re-renders the idle button bar. This is the correct approach for deferred focus — the button must exist in the DOM before `.focus()` is called. The "Delete selected" trigger is a logical return target in both cases. WCAG 2.4.3 satisfied.

### Finding 3: Cancel button has a descriptive accessible name — PASS

The Cancel button renders with visible text "Cancel" and no `aria-label` override. "Cancel" accurately describes the action (dismiss the confirmation without deleting). WCAG 4.1.2 (Name, Role, Value) satisfied.

### Finding 4: Delete button has a descriptive accessible name — PASS

The destructive confirmation button renders with visible text "Delete". In context — immediately following the confirmation prompt "Delete {selectedCount} conversation(s)?" — this is unambiguous. A screen reader user encountering this button in the confirm-delete state has already heard (or can navigate to) the confirmation prompt text.

If this component were audited in isolation (no surrounding context), "Delete" alone would be advisory — it lacks specificity about what is being deleted. In full context with the immediately preceding `<p>` element, it is acceptable. No action required.

### Finding 5: Focus indicator visibility on confirm-delete buttons — ADVISORY

The Cancel and Delete confirmation buttons carry these Tailwind classes:

**Cancel:**
```
flex-1 py-1 rounded text-[11px] text-text-secondary bg-hover hover:bg-hover/80 transition-colors duration-fast
```

**Delete:**
```
flex-1 py-1 rounded text-[11px] text-white bg-error-bg hover:opacity-90 transition-opacity duration-fast
```

Neither button has `focus-visible:ring-*` classes. The project's standard focus indicator pattern across all other interactive elements is:

```
focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-focus focus-visible:ring-offset-1
```

When the `useEffect` fires `confirmCancelRef.current?.focus()`, focus is applied programmatically — but because the Cancel button has no `focus-visible` ring classes, no visible focus indicator will appear in browsers that suppress the default `:focus` outline (which all modern browsers do by default for programmatically focused elements unless `tabindex` or keyboard navigation is in play). In practice, programmatic `.focus()` calls do not trigger `focus-visible` in Chrome, Firefox, or Safari — meaning the Cancel button will receive focus invisibly.

WCAG 2.4.7 (Focus Visible) requires a visible focus indicator for keyboard-operable elements. Although the focus is applied correctly, it is not visible to the user. This is a violation.

**Severity:** Serious — the focus is managed correctly but is invisible, defeating the benefit of the focus management fix for keyboard users who rely on the focus indicator to know where they are.

**WCAG criterion:** 2.4.7 (Focus Visible), Level AA.

**Recommendation for Aria:** Add the project-standard focus ring classes to both confirmation buttons in BulkActionBar. Example for Cancel:
```
className="flex-1 py-1 rounded text-[11px] text-text-secondary bg-hover hover:bg-hover/80 transition-colors duration-fast focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-focus focus-visible:ring-offset-1"
```
And for Delete:
```
className="flex-1 py-1 rounded text-[11px] text-white bg-error-bg hover:opacity-90 transition-opacity duration-fast focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-focus focus-visible:ring-offset-1"
```

Note: The idle-state "Archive selected" and "Delete selected" buttons in the same component also lack focus-visible ring classes (lines 843–862). This is the same pattern repeated across the entire BulkActionBar. Ada recommends fixing all four buttons in the same pass.

**Verdict: ADVISORY** — focus management logic is correct; focus visibility is missing on the confirmation buttons. This does not block the focus-routing fix from shipping, but should be remedied before the next Aria session. File as a follow-up issue.

## Issue #134 — Streaming shimmer: 4 new model variants

The `@media (prefers-reduced-motion: reduce)` block at line 247 contains explicit overrides for all 4 new variants:

```css
.streaming-shimmer[data-model="gemini"]::after {
  background: var(--accent-gemini);
  animation: none;
}

.streaming-shimmer[data-model="grok"]::after {
  background: var(--accent-grok);
  animation: none;
}

.streaming-shimmer[data-model="deepseek"]::after {
  background: var(--accent-deepseek);
  animation: none;
}

.streaming-shimmer[data-model="mistral"]::after {
  background: var(--accent-mistral);
  animation: none;
}
```

Each variant: (a) cancels the `streamingShimmer` keyframe animation via `animation: none`, and (b) collapses the gradient to a solid color per the model accent, matching the existing claude and gpt reduced-motion behavior. The pattern is consistent and complete. WCAG 2.3.3 (Animation from Interactions, AAA) is satisfied at the AA level by the presence of reduced-motion handling.

### Finding 2: No aria-label or role issues — PASS

The shimmer is a pure CSS `::after` pseudo-element applied via a class on a parent container. It carries no semantic weight and introduces no new ARIA attributes or roles. The pseudo-element is not exposed in the accessibility tree (pseudo-elements are never part of the DOM and cannot be referenced by ARIA). No accessibility attributes were introduced or removed.

### Finding 3: Shimmer pattern is consistent with existing variants — PASS

The 4 new variants follow exactly the same structure as the existing `claude` and `gpt` variants: animated gradient in the normal state, solid accent color with `animation: none` in the reduced-motion state. The CSS variable references (`--accent-gemini`, `--accent-grok`, `--accent-deepseek`, `--accent-mistral`) are confirmed defined in the `:root` block at lines 21–24 of index.css.

**Verdict: PASS**

## Issue #193 (follow-on) — MessageBubble: markdown link class swap to `text-prose-link`

**File:** `src/ui/MessageBubble.tsx` line 155

### Finding 1: `text-prose-link` is not a registered Tailwind token — BLOCKER

The link renderer in MessageBubble was changed from:
```
text-text-primary underline
```
to:
```
text-prose-link underline-offset-2
```

The Tailwind configuration at `tailwind.config.js` does not include `'prose-link'` in its `theme.extend.colors` map. The registered colors are: `bg`, `card`, `sidebar`, `input`, `text-primary`, `text-secondary`, `text-muted`, `text-inverse`, `border`, `border-subtle`, `border-strong`, `accent-claude`, `accent-gpt`, `accent-gemini`, `accent-other`, `accent-grok`, `accent-deepseek`, `accent-mistral`, `hover`, `active`, `focus`, `success`, `warning`, `error`, `error-bg`, `info`. The token `prose-link` is absent.

Tailwind v3's JIT compiler generates CSS only for tokens it finds in `theme.colors` or `theme.extend.colors`. The class `text-prose-link` does not produce a CSS rule in the compiled output — confirmed by searching the compiled `dist/assets/index-CCKnPpxW.css`, which contains no `prose-link` rule.

Additionally, even if the CSS were generated, `theme.ts` does not call `root.style.setProperty('--prose-link', ...)`. The CSS variable `--prose-link` is never set at runtime. The Luma spec in `_design/specs/tailwind-mapping.md` specifies the token as `colors.link` (Tailwind key) mapping to `--prose-link` (CSS variable). The correct Tailwind class is `text-link`, not `text-prose-link`. Aria used the wrong class name.

The net effect: the link receives no color from the prose-link token system. It falls back to inherited text color from the parent element.

**WCAG criterion:** 1.4.1 (Use of Color), Level A (required at AA).

**Severity:** Blocker — the link token is completely non-functional.

### Finding 2: Underline was removed — BLOCKER

The old implementation included the Tailwind class `underline`, which applies `text-decoration-line: underline`. The new implementation replaced it with `underline-offset-2`, which applies only `text-underline-offset: 0.5rem` — this sets the gap between the text baseline and an underline, but it does not apply the underline itself.

Without the `underline` class, markdown links in message bubbles have no underline decoration. They are differentiated from surrounding body text only by their color — which, per Finding 1, is non-functional. Links are currently visually indistinguishable from body text in message bubbles.

**WCAG criterion:** 1.4.1 (Use of Color), Level A (required at AA). The criterion states that color must not be the only visual means of conveying information. For links in body text, an underline (or other non-color visual indicator) is required. WCAG technique G183 (Using a contrast ratio of 3:1 with surrounding text, and providing additional visual cues on focus) is the alternative, but requires explicit implementation. The current state satisfies neither G183 nor the underline requirement.

**Severity:** Blocker — links in message bubbles are not distinguishable from body text by any non-color means.

### Finding 3: External link screen reader announcement is preserved — PASS

The external link `<span className="sr-only"> (opens in new tab)</span>` is still present at line 158. Screen reader users are still informed when a link opens in a new tab. This was not affected by the token swap.

### Finding 4: Focus indicator is preserved — PASS

The `focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-focus focus-visible:ring-offset-1 rounded-sm` classes remain on the anchor element. Keyboard focus will still produce a visible ring indicator. WCAG 2.4.7 satisfied.

### What the correct implementation looks like

The Luma spec (`_design/specs/tailwind-mapping.md` line 121, `_design/specs/markdown.md` line 62) specifies:

1. Tailwind token: `colors.link` → CSS variable `--prose-link`
2. Tailwind class: `text-link` (not `text-prose-link`)
3. Underline: "Decoration: underline always applied to links via Tailwind class"

The correct fix requires three coordinated changes:

1. **Aria:** Register `'link': 'var(--prose-link)'` and `'link-hover': 'var(--prose-link-hover)'` in `tailwind.config.js` `theme.extend.colors`
2. **Aria:** Update `src/ui/theme.ts` to call `root.style.setProperty('--prose-link', theme.prose.link)` and `root.style.setProperty('--prose-link-hover', theme.prose['link-hover'])`
3. **Aria:** Update `MessageBubble.tsx` line 155 to use `text-link underline` instead of `text-prose-link underline-offset-2`

The `underline-offset-2` class may be kept alongside `underline` for better visual spacing (it is a valid refinement), but `underline` itself must be present.

Until these fixes land, markdown links in message bubbles display as unstyled body text with no color or underline differentiation.

**Verdict: BLOCKED** — two WCAG 1.4.1 failures. The `text-prose-link` token is non-functional and the `underline` decoration was dropped. Both must be resolved before this branch merges.

### WCAG 1.4.1 assessment

The fix took a different path than the one Ada prescribed. Rather than wiring the `link` token through tailwind.config.js and theme.ts, Aria reverted the link class to `text-text-primary underline`. This resolves both original blockers under WCAG 1.4.1:

- **Finding 2 (underline dropped) — RESOLVED.** `underline` is present. Links carry `text-decoration-line: underline`. A non-color visual differentiator exists.
- **Finding 1 (token non-functional) — RESOLVED by different means.** `text-text-primary` is a registered, functional Tailwind token that resolves to a real CSS custom property set in `applyTheme()`. The link color matches body text, but WCAG 1.4.1 does not require links to be a distinct color — it requires that color not be the *only* means of conveying the link status. The underline satisfies that requirement independently.

The approach is a valid interim solution. The `text-link` token path remains a TODO for a future Luma/Aria session, but it is not required for WCAG 1.4.1 compliance in this state.

### Finding: PASS

Both original blockers are resolved. WCAG 1.4.1 (Use of Color) is satisfied by the combination of `text-text-primary` (functional color) and `underline` (non-color differentiator).

### Updated overall verdict: PASS
