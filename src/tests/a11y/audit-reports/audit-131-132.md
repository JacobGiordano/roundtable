## InteractionModeSwitcher ŌĆö #131

### Audit question 1: Can keyboard users reach the disabled spans?

**PASS.** Manual and Auto-chain are rendered as `<span>` elements with no `tabIndex` attribute. Spans are not focusable by default in HTML. The automated test (`interaction-mode-switcher.test.tsx`) confirms no `tabIndex` is set, and `queryAllByRole('radio')` returns exactly one element (Parallel), confirming the disabled spans are not reachable via keyboard or screen reader role navigation.

### Audit question 2: Does the tooltip communicate the "coming soon" state to screen readers?

**ADVISORY.** The tooltip is hover-only (`opacity-0 group-hover:opacity-100`). It has `role="tooltip"` correctly applied, but there is no `aria-describedby` connecting the span to the tooltip element. This means a screen reader user reading in browse/virtual cursor mode will hear the `aria-label` on the span ("Manual ŌĆö coming soon" / "Auto-chain ŌĆö coming soon"), which does communicate the state ŌĆö but the richer tooltip text ("Coming soon ŌĆö not yet available") is hover-only and invisible to assistive technology.

The `aria-label` on the span carries the essential information and is a reasonable mitigation. However, the WAI-ARIA tooltip pattern (APG) specifies that the triggering element should reference the tooltip via `aria-describedby`. Without it, the tooltip `role="tooltip"` has no programmatic connection to anything and is essentially orphaned in the accessibility tree.

This is not a WCAG AA failure because the span is non-interactive (WCAG 1.4.13 applies to "additional content on hover or focus" for interactive elements ŌĆö a non-interactive span is exempt from that criterion). The `aria-label` provides the necessary non-color indicator. However, connecting the tooltip to its element is correct practice.

**Filed as advisory.** See ticket recommendation below.

### Audit question 3: Is the radiogroup ARIA semantics coherent with mixed button/span children?

**ADVISORY ŌĆö warrants clear-eyed consideration.**

The `role="radiogroup"` container now has children of two fundamentally different types: one `<button role="radio">` (Parallel) and two plain `<span>` elements (Manual, Auto-chain). Per WAI-ARIA 1.2, `radiogroup` is an owned container for `radio` role elements. Spans with no role are not `radio` descendants ŌĆö they are arbitrary content inside a `radiogroup`.

In practice:
- `aria-required-owned-elements` for `radiogroup` is `radio` ŌĆö AT implementations vary in how strictly they enforce this. Most screen readers (NVDA, JAWS, VoiceOver) will announce the group label and the one `role="radio"` button correctly, then read the spans as static text as the cursor passes over them.
- The spans carry `aria-label="[Mode] ŌĆö coming soon"` so their content is communicated; they are just not announced as radio options.
- NVDA in Forms mode will skip the spans entirely (correct behavior ŌĆö they are not form controls). VoiceOver may read them in browse mode.

There is no WCAG AA violation here because non-interactive spans inside a `radiogroup` do not create an accessibility failure ŌĆö they are simply inert content inside a landmark. The semantics are coherent enough for AT to use. The `radiogroup` has exactly one selectable `radio` child, which is the current functional behavior.

The cleaner long-term pattern would be to give the spans `role="radio" aria-disabled="true"` with `aria-checked="false"` ŌĆö this keeps the ownership model pure, puts them in the radio keyboard navigation cycle (where arrow keys skip disabled radios), and communicates their disabled state via standard ARIA rather than custom `aria-label` text. However, this is an enhancement, not a WCAG AA failure.

**Filed as advisory.** See ticket recommendation below.

### Audit question 4: Does the disabled visual state (opacity-50) meet contrast requirements?

**PASS ŌĆö WCAG disabled component exception applies intentionally.**

The disabled spans render as `text-text-muted opacity-50`. The computed effective contrast after blending:

| Theme | text-muted | bg-sidebar | Blended effective | Ratio |
|-------|-----------|------------|-------------------|-------|
| Slate | `#7C84A2` | `#13151C` | `#484D5F` | **2.17:1** |
| Linen | `#6D6863` | `#EDE8DF` | `#ADA8A1` | **1.94:1** |

These ratios are well below the 4.5:1 text minimum and also below the 3:1 UI component minimum. However, WCAG 1.4.3 explicitly exempts disabled components: "Text or images of text that are part of an inactive user interface component... have no contrast requirement."

The spans are genuinely disabled ŌĆö they cannot be activated, they exist only as visual placeholders, and the visual muting (opacity-50) directly communicates the disabled state. The exception applies as intended. The design is using the exemption correctly, not exploiting it to hide a contrast deficiency on interactive content.

One note: if these spans were ever re-enabled (i.e., the `comingSoon` flag removed) but the `opacity-50` styling was accidentally left in place, contrast would fail. This is a code maintenance risk, not a current violation.

## MessageBubble ŌĆö #132

### Audit question 1: Links ŌĆö target/rel and new-tab announcement

**PASS ŌĆö target/rel correct. ADVISORY ŌĆö no new-tab announcement for screen readers.**

The `a` renderer (line 145ŌĆō157) correctly applies `target="_blank" rel="noopener noreferrer"` to external links. The automated test confirms this.

However, the link does not communicate "opens in new tab" to screen readers. WCAG 2.4.4 (Link Purpose) requires that link purpose is determinable from the link text alone or from link text plus programmatic context. Opening in a new tab is a behavior change that many users ŌĆö especially screen reader users ŌĆö consider important to know in advance (they cannot use the browser Back button to return from a new tab in some workflows).

Best practice is either:
- Append visually hidden text: `<span className="sr-only"> (opens in new tab)</span>` inside the anchor, or
- Extend the `aria-label`: `aria-label={`${linkText} (opens in new tab)`}`

Without this, a link that reads "See the documentation" gives no indication that activation will open a new tab rather than navigate the current page. This is a frequent screen reader usability complaint.

WCAG 2.4.4 is PASS (link purpose is determinable from link text in most cases). The new-tab behavior is a WCAG 3.2 (Predictable) consideration ŌĆö WCAG 3.2.2 (On Input) does not strictly require announcement, but G201 (giving users advanced warning when opening a new window) is a sufficient technique for 3.2.2 conformance.

**Filed as advisory.** See ticket recommendation below.

### Audit question 2: Heading hierarchy inside message bubbles

**ADVISORY ŌĆö unconstrained `<h1>` is a page structure problem.**

The `h1` renderer (line 107ŌĆō108) renders markdown `#` as a native `<h1>`. Inside a message bubble that lives within a `<main>` element with a page-level heading structure, any `<h1>` generated from model output will collide with the page's own heading hierarchy. A screen reader user navigating by headings will encounter `<h1>` elements mid-conversation as if they were page titles.

The markdown heading levels should be downshifted by at least 2 levels inside conversation content:
- `#` (h1 in markdown) ŌåÆ `<h3>` in DOM
- `##` ŌåÆ `<h4>`
- `###` ŌåÆ `<h5>`
- `####` ŌåÆ `<h6>`
- `#####` and `######` ŌåÆ `<h6>` or `<p className="font-semibold">`

This is a semantic HTML correctness problem with measurable screen reader impact (WCAG 1.3.1 Info and Relationships ŌĆö heading levels convey document structure). It is not an axe violation (axe does not flag heading levels as incorrect purely from the presence of `<h1>`) but it is a genuine WCAG 1.3.1 concern.

**Filed as advisory ŌĆö but close to blocker.** If the page has an `<h1>` or `<h2>` in `AppLayout.tsx`, a model response containing `# My Section` will outrank it in the heading outline. This should be fixed before the feature is considered production-ready.

### Audit question 3: Code blocks ŌĆö screen reader accessibility

**PASS.** Code blocks render as `<pre><code>` (lines 182ŌĆō188), which is the correct accessible structure. The `<code>` element is present as a child of `<pre>`. Screen readers announce pre/code elements and read the content linearly. There is no WAI-ARIA role needed on code blocks ŌĆö the semantic HTML is sufficient.

One observation: the code block has no language label. When react-markdown passes `className="language-typescript"`, the inner `code` element carries that class but it is not surfaced as visible text or an ARIA label. A user who cannot see the code block has no way to know what language it is in. This is a minor gap ŌĆö language identification is not a WCAG AA requirement, but it would improve the experience for screen reader users.

**No ticket needed for this observation ŌĆö noted for completeness only.**

### Audit question 4: aria-live / aria-atomic ŌĆö not conflicted by markdown output

**PASS.** The `aria-live="polite"` and `aria-atomic="false"` attributes are on the wrapper `div` that contains the `ReactMarkdown` component (lines 241ŌĆō255), not on any element rendered by react-markdown itself. The markdown renderers (`p`, `h1`ŌĆō`h6`, etc.) do not add any live region attributes. There is no conflict. The streaming live region behavior is preserved correctly.

The automated tests in both `message-bubble.test.tsx` (pre-existing) and `message-bubble-markdown.test.tsx` (new) confirm the live region attributes are correct in both streaming and completed states.

### ADVISORY

**[A1] InteractionModeSwitcher ŌĆö tooltip not connected via `aria-describedby`**
- Component: `src/ui/InteractionModeSwitcher.tsx` (lines 91ŌĆō132, disabled span branch)
- Issue: The `<div role="tooltip">` has no `id`, and the triggering span has no `aria-describedby`. The tooltip is orphaned in the ARIA tree. Screen readers that support tooltip roles may not associate it with the span.
- WCAG criterion: Adjacent to **1.3.1 Info and Relationships** ŌĆö the tooltip provides information but the relationship is not programmatically determinable.
- Recommended fix: Add `id="tooltip-manual"` / `id="tooltip-auto-chain"` to each tooltip div; add `aria-describedby="tooltip-manual"` to the corresponding span. Note: tooltip visibility must also respond to keyboard focus (not just hover) for interactive elements ŌĆö for these non-interactive spans, hover-only is acceptable.
- Severity: Moderate (the `aria-label` on the span already communicates the coming-soon state; this is a structural improvement, not a functional gap)
- Owner: Aria

**[A2] InteractionModeSwitcher ŌĆö mixed `role="radiogroup"` children (button + span)**
- Component: `src/ui/InteractionModeSwitcher.tsx` (lines 216ŌĆō235)
- Issue: `role="radiogroup"` expects `role="radio"` owned children. Spans with no role break the ownership model. While AT behavior is acceptable in practice, the semantics would be cleaner with `role="radio" aria-disabled="true"` on the coming-soon spans.
- WCAG criterion: Adjacent to **4.1.2 Name, Role, Value** ŌĆö the ownership model is non-standard.
- Recommended fix: When enabling Manual/Auto-chain, remove the span branch entirely and restore `<button role="radio">` with `disabled` or `aria-disabled="true"`. Until then, consider adding `role="radio" aria-disabled="true" aria-checked="false"` to the spans to keep the radiogroup ownership clean.
- Severity: Minor (current AT behavior is acceptable; this is a structural improvement for future maintainability)
- Owner: Aria

**[A3] MessageBubble ŌĆö external links do not announce "opens in new tab" to screen readers**
- Component: `src/ui/MessageBubble.tsx` (lines 145ŌĆō157, `a` renderer)
- Issue: `target="_blank"` links open in a new tab without prior announcement. Screen reader users may not anticipate this behavior change.
- WCAG criterion: Adjacent to **3.2.2 On Input** (G201 sufficient technique) and **2.4.4 Link Purpose**
- Recommended fix: Append `<span className="sr-only"> (opens in new tab)</span>` inside external `<a>` elements.
- Severity: Moderate
- Owner: Aria

**[A4] MessageBubble ŌĆö heading levels not downshifted; `<h1>` in message content disrupts page outline**
- Component: `src/ui/MessageBubble.tsx` (lines 107ŌĆō123, `h1`ŌĆō`h6` renderers)
- Issue: Markdown `#` renders as native `<h1>` in the DOM. Model-generated `<h1>` elements inside `<main>` disrupt the page heading hierarchy and confuse screen reader users navigating by heading.
- WCAG criterion: **1.3.1 Info and Relationships** ŌĆö heading levels convey document structure; allowing `<h1>` from model output breaks the document outline.
- Recommended fix: Downshift all heading levels by 2: `h1 ŌåÆ h3`, `h2 ŌåÆ h4`, `h3 ŌåÆ h5`, `h4 ŌåÆ h6`, `h5/h6 ŌåÆ h6` or styled `<p>`. Update the six renderer functions in `markdownComponents` accordingly.
- Severity: Serious (heading navigation is a primary screen reader navigation pattern; unconstrained `<h1>` from model content is a persistent structural problem in every conversation)
- Owner: Aria
