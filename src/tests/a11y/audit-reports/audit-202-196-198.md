## Issue #202 — AppLayout: Persistent `<h1>` in `<main>`

**File:** `src/ui/AppLayout.tsx` line 270

### Finding 1: h1 unconditional placement — PASS

The `<h1 className="sr-only">Roundtable Conversation</h1>` is placed as the first child of `<main>`, before the conditional that branches between `<OnboardingEmptyState>` and `<MessageThread>`. It is present in the DOM regardless of which branch renders. The comment confirms intent: "Unconditional: present whether OnboardingEmptyState is showing or not." This is correct.

### Finding 2: `sr-only` utility class — PASS

`sr-only` is a first-class Tailwind CSS v3 utility. It renders as:

```css
position: absolute;
width: 1px;
height: 1px;
padding: 0;
margin: -1px;
overflow: hidden;
clip: rect(0, 0, 0, 0);
white-space: nowrap;
border-width: 0;
```

This is the WCAG-standard technique for visually hiding content while keeping it in the accessibility tree (WCAG 1.3.1). The element remains readable by screen readers and navigable via heading shortcuts. Correct choice.

### Finding 3: Heading hierarchy — ADVISORY

There is a second `<h1>` in the page: `OnboardingEmptyState.tsx` line 92 renders `<h1>Welcome to Roundtable</h1>` with visible styling. Because the AppLayout's `<h1>` is unconditional and `OnboardingEmptyState` renders conditionally inside `<main>`, the two `<h1>` elements coexist in the DOM when `isRosterEmpty` is true.

WCAG 1.3.1 (Info and Relationships) does not formally prohibit multiple `<h1>` elements, and WCAG 2.4.6 (Headings and Labels) only requires that headings describe the topic. However, the WAI-ARIA Authoring Practices and the HTML specification both recommend a single `<h1>` per page as the document's primary title. Two `<h1>` elements at the same level fragment the heading landmark for screen reader users navigating by heading, and some screen readers (notably JAWS in "legacy" document mode) give elevated prominence to the first `<h1>` and treat subsequent ones as structurally ambiguous.

The existing `OnboardingEmptyState` `<h1>` predates this fix and was already a pre-existing issue. The fix itself (AppLayout's `<h1>`) is technically correct in isolation — a persistent heading in `<main>` is the right pattern. But the combination now produces two `<h1>` elements on first load.

**Severity:** Advisory — not a blocker for this fix, but worth resolving in a follow-up.

**Recommendation for Aria:** Downshift `OnboardingEmptyState.tsx` line 92 from `<h1>` to `<h2>`. The AppLayout `<h1>` "Roundtable Conversation" is the correct page-level landmark; the onboarding heading "Welcome to Roundtable" is a section heading within that landmark. This would give the page a clean single `<h1>` in all states.

### Finding 4: Heading text is meaningful — PASS

"Roundtable Conversation" accurately describes the content region. Screen readers reading the heading outline will encounter it when navigating to the `<main>` landmark. It is more informative than a generic "Main content" label. WCAG 2.4.6 satisfied.

### Finding 5: No other `<h1>` in the page outside OnboardingEmptyState — PASS

Confirmed by grep across `/src/ui/`: the only `<h1>` elements are the AppLayout `sr-only` heading (line 270) and the OnboardingEmptyState heading (line 92). No other component introduces an `<h1>`. The message content headings in `MessageBubble.tsx` are correctly downshifted starting at `<h3>` per #201.

## Issue #196 — Sidebar: Ghost Mode Toggle aria-label

**File:** `src/ui/Sidebar.tsx` lines 1232–1259

### Finding 1: Static aria-label + aria-pressed pattern — PASS

The ghost mode toggle button is implemented as:

```html
<button
  aria-label="Ghost mode"
  aria-pressed={isGhostMode}
  ...
>
```

This is the correct WAI-ARIA toggle button pattern per the WAI-ARIA Authoring Practices Guide (APG), section "Button Pattern — Toggle Button." The APG states: "When the button is activated, its `aria-pressed` state is toggled. The label should remain constant; the state change is communicated via `aria-pressed`."

A static label with `aria-pressed` is preferable to a dynamic label that embeds state (e.g., "Ghost mode on — click to disable") because:

1. Screen readers announce the `aria-pressed` state separately and consistently, giving users a predictable announcement pattern.
2. Dynamic labels that change on activation can confuse users who lose focus context — they hear a different button name after the action completes.
3. The pattern matches the established ARIA button toggle contract that assistive technologies already know how to announce.

VoiceOver on macOS will announce: "Ghost mode, toggle button, [on/off]" — clear and unambiguous. NVDA produces a similar announcement. WCAG 4.1.2 (Name, Role, Value) satisfied.

### Finding 2: State not leaked into any other attribute — PASS

No other attribute on the button embeds the ghost mode state. The button's inner text is empty (SVG only, which is `aria-hidden="true"`). No `title` attribute is present. No `aria-description` or `aria-describedby` encodes the state. State communication is exclusively via `aria-pressed`. Correct.

### Finding 3: aria-pressed value changes with toggle state — PASS

`aria-pressed={isGhostMode}` is bound directly to the prop. React will render `aria-pressed="true"` when `isGhostMode` is `true` and `aria-pressed="false"` when `isGhostMode` is `false`. This is the correct behavior. (The test stubs in `sidebar-ghost-toggle.test.tsx` cover both states and should be activated now that #196 has shipped.)

### Finding 4: Focus ring visible — PASS

The button carries:
```
focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-focus focus-visible:ring-offset-2
```
This is the project's standard focus indicator pattern, consistent with all other buttons in the sidebar. WCAG 2.4.7 (Focus Visible) satisfied.

### Finding 5: Touch target size — PASS

The button is `w-8 h-8` (32×32px). This is below the WCAG 2.5.5 (Target Size) AAA minimum of 44×44px, but WCAG 2.5.5 is AAA — not required at AA. At AA, the criterion is 2.5.8 (Target Size — Minimum, WCAG 2.2 addition at AA level) which requires 24×24px. 32×32px satisfies this. Noted as AAA shortfall only.

## Issue #198 — InteractionModeSwitcher: Tooltip aria-describedby Wiring

**File:** `src/ui/InteractionModeSwitcher.tsx` lines 86–187

### Finding 1: Tooltip IDs are unique per mode — PASS

`tooltipId` is computed as `` `interaction-mode-tooltip-${config.mode}` `` (line 86). The three modes are `parallel`, `manual`, and `auto-chain`, producing:

- `interaction-mode-tooltip-parallel`
- `interaction-mode-tooltip-manual`
- `interaction-mode-tooltip-auto-chain`

All three are distinct. The IDs are stable (derived from the enum value, not from a runtime counter), which means they are predictable in tests and consistent across renders. If `InteractionModeSwitcher` were mounted multiple times on a page, the IDs would collide — but this component is rendered exactly once in `AppLayout`, so this is not a practical concern. PASS.

### Finding 2: Disabled spans have aria-describedby pointing to tooltip — PASS

For the disabled (coming-soon) case (lines 96–109):

```html
<span
  aria-label="Manual — coming soon"
  aria-describedby="interaction-mode-tooltip-manual"
>
```

The corresponding tooltip `<div id="interaction-mode-tooltip-manual" role="tooltip">` exists in the same DOM subtree (lines 113–134). The `aria-describedby` reference is valid and will resolve correctly. When a screen reader encounters this span in reading/browse mode (it is not keyboard-focusable, so it is only reachable via reading), the description will be provided by the tooltip text: "Coming soon — not yet available."

### Finding 3: Active buttons have aria-describedby pointing to tooltip — PASS

For the active (selectable) case (lines 141–161):

```html
<button
  role="radio"
  aria-checked={isSelected}
  aria-label="Parallel — All active models respond simultaneously to every message."
  aria-describedby="interaction-mode-tooltip-parallel"
>
```

The `aria-label` already contains the description text, and the `aria-describedby` points to the tooltip which contains the same description. This creates redundant announcement — when the button receives focus, a screen reader will typically announce the `aria-label` first, then the `aria-describedby` text, producing a doubled description.

This is a minor UX consideration, not a WCAG failure. Both `aria-label` and `aria-describedby` carry meaningful content; the redundancy is harmless and arguably reinforces comprehension. The `aria-label` ensures the button has an accessible name regardless of whether the tooltip is associated. The `aria-describedby` ensures keyboard users who hover (or whose AT exposes tooltip relationships) also receive the description.

**Severity:** Minor advisory. The doubled announcement is not disorienting — users can skip it with screen reader navigation shortcuts. No action required before ship.

### Finding 4: Tooltip role="tooltip" semantic correctness — PASS

`role="tooltip"` is the correct ARIA role for a short descriptive element associated via `aria-describedby`. The tooltip elements are positioned via CSS (opacity-0/opacity-100, pointer-events-none) and do not receive focus. They are persistent in the DOM (not inserted/removed), which is the recommended pattern for `role="tooltip"` elements as it allows `aria-describedby` to reference them reliably.

Note: The tooltip text for active buttons is always visible in the DOM (just opacity-0), which means AT can read it via the `aria-describedby` reference even when it is not visually shown. This is the correct behavior.

### Finding 5: The radiogroup owns the coming-soon spans — ADVISORY

The `<div role="radiogroup">` contains both the interactive `<button role="radio">` (Parallel) and the non-interactive `<span>` elements (Manual, Auto-chain) wrapped in `<div class="relative group">` containers. The ARIA radiogroup ownership model expects only radio children as owned elements. Non-radio children inside a radiogroup are technically owned by the group but are not recognized as group members by AT.

This is a pre-existing structural issue filed as #199 ("Coming-soon spans break radiogroup ownership") and is out of scope for this fix. Issue #198 only adds `aria-describedby` wiring. The wiring itself does not worsen the ownership issue and is correct within the existing structure. The #198 fix should ship.

### Finding 6: Tooltip text is meaningful — PASS

- Parallel: "All active models respond simultaneously to every message." — accurately describes the mode.
- Manual: "Coming soon — not yet available" — accurately communicates why the mode is unavailable.
- Auto-chain: "Coming soon — not yet available" — same.

All three tooltip texts are informative to screen reader users. WCAG 2.4.6 satisfied.
