## Serious Findings

### G1 — ProviderRow confirm states: focus not moved to Cancel button on state transition
**File**: `src/ui/ProviderSettingsPanel.tsx` — `ProviderRow`, lines 107–426
**WCAG**: 2.4.3 — Focus Order
**GitHub**: #115
**Affects**: Both `confirm-remove` (existing) and `confirm-remove-last` (new in #108)
**Testing method**: Static source analysis

When a keyboard user presses Enter or Space on the trash (Remove) button in a `ProviderRow`, `handleRemoveClick` fires and `setConfirmState` transitions the row from `'idle'` to either `'confirm-remove'` or `'confirm-remove-last'`. The previously focused trash button is no longer in the DOM. No `useRef`/`useEffect` pattern moves focus to the Cancel or Remove button in the new confirmation UI. The browser moves focus to `document.body`.

The `aria-live="polite"` on the listitem container (`line 198`) correctly announces the new confirmation text to screen readers — this is good. But the user must Tab from wherever the browser placed focus (potentially from the top of the document) back into the settings panel to reach the Cancel or Remove buttons.

Issue #108 introduced the `confirm-remove-last` state, which has the same focus management gap as the pre-existing `confirm-remove` state. Both should be fixed together.

**Impact**: A keyboard-only user who triggers the remove confirmation cannot reach the Cancel or Remove buttons without extensive re-navigation.

**Fix (for Aria)**:
```tsx
const cancelBtnRef = useRef<HTMLButtonElement>(null);

useEffect(() => {
  if (confirmState !== 'idle') {
    requestAnimationFrame(() => cancelBtnRef.current?.focus());
  }
}, [confirmState]);
```
Attach `ref={cancelBtnRef}` to the Cancel button in both `confirm-remove` (line 364) and `confirm-remove-last` (line 399) branches.

### G2 — Provider settings drawer: no focus trap; keyboard can reach content behind the overlay
**File**: `src/ui/AppLayout.tsx:220–228` + `src/ui/ProviderSettingsPanel.tsx:849–850`
**WCAG**: 2.1.2 — No Keyboard Trap (inverse: the overlay has no containment); 2.4.3 — Focus Order
**GitHub**: #116
**Affects**: #111 (backdrop) + pre-existing panel configuration
**Testing method**: Static source analysis

The provider settings drawer renders with `role="dialog"` and `aria-modal="false"` (`ProviderSettingsPanel.tsx:849–850`). Issue #111 added a backdrop (`AppLayout.tsx:220–228`) that visually dims the page and intercepts mouse clicks, giving the appearance of modal behavior. However:

1. **No keyboard focus containment**: `aria-modal="false"` and no JavaScript focus trap means keyboard users can Tab past the drawer's last focusable element and reach the chat input, message thread, model selector, and other controls behind the overlay.

2. **Screen readers are not told background is inert**: `aria-modal="false"` allows AT to read and navigate the full page tree. Background conversation controls are accessible to screen reader users even when the settings drawer is open.

3. **Sidebar accessible via keyboard on desktop**: The backdrop at `fixed inset-0 z-30` intercepts mouse events over the entire viewport including the sidebar area — but keyboard Tab follows DOM order, not visual z-index. Sidebar controls (thread rows, settings toggle, new conversation button) are reachable via keyboard while the provider settings drawer is open.

The visual treatment (dimmed backdrop, shadow, slide-in) communicates "this is modal" to sighted users. The keyboard and AT behavior contradicts this presentation.

**Fix (for Aria)**:

**Option A — True modal (recommended)**: Change `aria-modal="false"` to `aria-modal="true"` and implement a focus trap:
```tsx
// When isOpen is true, trap focus within the panel container.
// On Tab: if last focusable element, wrap to first.
// On Shift+Tab: if first focusable element, wrap to last.
```
The panel already has Escape handling, focus-on-open, and focus-return-on-close — the missing piece is Tab containment and `aria-modal="true"`.

**Option B — Non-modal drawer**: Remove `role="dialog"` (or replace with `role="complementary"`/`<aside>`). The backdrop remains a mouse-click convenience. Keyboard users can freely navigate while the drawer is open. Document this intent explicitly. This is a valid but significantly different interaction model.

Option A is recommended. The existing code architecture (close button focus, Escape handling, trigger ref focus return) follows the modal dialog pattern.

## Minor Findings

### G3 — Backdrop missing `motion-reduce:transition-none` (AAA, consistency issue)
**File**: `src/ui/AppLayout.tsx:222–225`
**WCAG**: 2.3.3 — Animation from Interactions (AAA — noted, not a AA failure)
**GitHub**: #117
**Testing method**: Static source analysis of `AppLayout.tsx` + `src/index.css`

The backdrop uses Tailwind `transition-opacity duration-200` but does not include `motion-reduce:transition-none`. By contrast, the drawer itself (`ProviderSettingsPanel.tsx:855`) correctly applies `motion-reduce:transition-none`. The codebase is otherwise meticulous: `pill-shake`, `chunk-entering`, `cursor-blink`, `bubble-entering`, `thread-entering`, `streaming-shimmer`, the model-selector-panel transition, and `provider-enter` all have `prefers-reduced-motion` overrides. The backdrop fade-in/out is the only unaddressed CSS transition.

WCAG 2.3.3 is AAA — this is not a AA failure. It is filed as Minor due to the inconsistency with the project's established reduced-motion policy and the intentional treatment of the drawer's own transition.

**Fix (for Aria)**: Add `motion-reduce:transition-none` to the backdrop className:
```tsx
'transition-opacity duration-200 motion-reduce:transition-none',
```

### G4 — Settings shortcut button: no signal that the model selector will close (advisory)
**File**: `src/ui/ModelSelectorPanel.tsx:1146–1174`
**WCAG**: 4.1.2 — Name, Role, Value (advisory, not a failure)
**GitHub**: #118
**Testing method**: Static source analysis

The "Provider settings" shortcut button at the bottom of the ModelSelectorPanel has `aria-label="Open provider settings"`, which correctly names the action. When clicked, `handleOpenSettingsFromPanel` both closes the model selector and opens the provider settings drawer. The accessible name does not communicate that the current panel will close as a side effect.

This is advisory, not a WCAG AA failure. The button is correctly labeled, has the correct role (`button`), and communicates its purpose. The cross-panel navigation behavior is a UX transparency concern.

The button's gear icon is correctly `aria-hidden="true"`. The visible text "Provider settings" provides an accessible name that is reinforced by the `aria-label`. No labeling failure.

**Suggestion (advisory)**: If the navigation surprise is a concern, consider amending the `aria-label` to `"Provider settings (closes model selector)"` or restructuring the interaction so the model selector does not close unexpectedly (e.g. open the provider settings panel without closing the model selector).

## Clean Findings — Confirmed Accessible

### #108 — confirm-remove-last: confirmation text is announced via aria-live
**PASS** — `ProviderRow`'s root `div` has `aria-live="polite"` (line 198). When `confirmState` changes from `'idle'` to `'confirm-remove-last'`, the new text content ("Last provider. Remove and start over?") is in the live region and will be announced by screen readers. This provides a partial mitigation for the focus management gap (G1) — users will hear the confirmation even though focus does not move to the buttons.

### #108 — Cancel and Remove buttons have correct semantics
**PASS** — Both buttons in the `confirm-remove-last` branch (lines 399–421) are native `<button type="button">` elements with visible text labels ("Cancel", "Remove"), focus rings (`focus-visible:ring-2 focus-visible:ring-focus focus-visible:ring-offset-1`), and correct button roles. No ARIA needed.

### #108 — Error accent on confirmation row communicates state redundantly
**PASS** — The confirmation row uses `borderLeftColor: var(--semantic-error)` (an accent, not text) alongside the warning text "Last provider. Remove and start over?". Color is NOT the only indicator — the text content communicates the warning state independently. WCAG 1.4.1 (Use of Color) satisfied.

### #109 — maxWidth: 704px on panel shell
**PASS** — Layout-only change. No interactive elements added or altered. The panel remains fully keyboard navigable, and the content body's `max-w-[640px]` constraint with `mx-auto` (#112) centers content correctly. No accessibility impact.

### #110 — Settings shortcut: button is keyboard-operable with visible focus ring
**PASS** — The button has `focus-visible:ring-2 focus-visible:ring-focus focus-visible:ring-offset-1 rounded`. It is natively keyboard-operable (native `<button>` element). No custom keydown handling needed.

### #110 — Settings shortcut: gear icon is aria-hidden
**PASS** — The gear SVG at lines 1159–1172 has `aria-hidden="true"`. The button's accessible name comes from the visible text "Provider settings" and the explicit `aria-label`. The icon is purely decorative in this context.

### #111 — Backdrop is correctly aria-hidden
**PASS** — The backdrop div has `aria-hidden="true"` (AppLayout.tsx:221). This prevents the backdrop from appearing in the accessibility tree and being announced by screen readers. Mouse-click event handler on an `aria-hidden="true"` element is an acceptable pattern for a click-outside-to-close interaction.

### #111 — Backdrop pointer-events-none when closed
**PASS** — `pointer-events-none` is applied when `isProviderPanelOpen` is false, preventing the invisible backdrop from intercepting mouse events when the drawer is closed.

### #111 — Escape key dismissal
**PASS** — Escape to close the drawer is implemented in `ProviderSettingsPanel.tsx:799–809` via `document.addEventListener('keydown', handleKeyDown)`. This is wired up before #111 shipped and remains intact.

### #111 — Focus return to trigger on backdrop click
**PASS** — `handleCloseProviderSettings` calls `onClose` which eventually calls `handleClose` in `ProviderSettingsPanel`, which returns focus to `triggerRef.current` via `requestAnimationFrame`. Whether the panel closes via Escape, close button, or backdrop click, focus returns to the sidebar gear button.

### #112 — mx-auto on panel body
**PASS** — Layout-only change. No interactive elements affected. No accessibility impact.

## Reduced-Motion Audit (Changes Only)

| Element | Motion | Reduced-motion override |
|---------|--------|------------------------|
| Drawer slide-in/out | CSS transform transition | `motion-reduce:transition-none` on panel ✓ |
| Backdrop opacity fade | CSS opacity transition | **Missing `motion-reduce:transition-none`** ✗ (G3) |
| Provider row enter animation | `@keyframes provider-enter` | Overridden in index.css media query ✓ |
| Provider row removal | CSS height+opacity transition (inline style) | Not covered by media query — inline style bypasses CSS media queries ⚠ |

Note on the inline-style row removal animation: `rowStyle` applies `height: 0, opacity: 0, transition: 'height 200ms ease-in, opacity 100ms ease-in'` via inline style. CSS `prefers-reduced-motion` media queries do not affect inline styles. This means reduced-motion users will still see the collapse animation when a provider row is removed. This is a low-severity gap but consistent with the pattern of inline style transitions elsewhere in the codebase (e.g. the drawer slide transition). Filed under this report for awareness; not opened as a separate issue given the minor impact.
