## Audit Findings

### 1. Heading semantics — PASS

**Finding**: The downshift mapping is correct. Analysis of the page heading outline:

- `<h1>` appears exclusively in `OnboardingEmptyState.tsx` ("Welcome to Roundtable"), which is rendered only when no providers are configured. It is never in the DOM simultaneously with message content.
- `<h2>` appears in `ProviderSettingsPanel.tsx` (`id="psp-heading"`) as the panel header inside a modal overlay (fixed, z-40). It is correctly scoped to that landmark context.
- `<main id="main-content">` in `AppLayout.tsx` contains `MessageThread` → `MessageBubble`. The first heading that can appear inside `<main>` from model-generated markdown is now `<h3>`. This is appropriate: the page shell occupies h1 (onboarding) and h2 (modal overlay), and the first heading inside the main content well starts at h3, preserving a logical outline without skipping levels.
- `h4`, `h5`, `h6` follow in order. The collapse of markdown `h4`/`h5`/`h6` → all `<h6>` creates a flat tail, but this is a documented trade-off: the HTML heading model has no lower level than `<h6>`, and flattening the tail is preferable to emitting `<h1>` or `<h2>` from model content.

**WCAG 1.3.1** (Info and Relationships): The semantic heading structure conveys the same information as the visual hierarchy for all mapped levels. Pass.

**Note**: When `OnboardingEmptyState` is not in the DOM (the normal post-onboarding state) and `ProviderSettingsPanel` is closed, the first heading in the page is `<h3>` inside `<main>`. This skips h1 and h2 in the main landmark. This is a pre-existing structural characteristic of the app (no persistent page-level `<h1>` exists when the conversation view is active). It is not introduced by this PR — it predates these changes. It warrants a future advisory ticket but is not a blocker for this diff.

### 2. `sr-only` span — PASS

**Finding**: The `<span className="sr-only"> (opens in new tab)</span>` at line 164 of `MessageBubble.tsx` will be announced by screen readers. Verified:

- The span is a direct child of the `<a>` element, inside the `a` renderer of `markdownComponents`.
- No `aria-hidden` attribute exists on the `<a>` element, the `markdownComponents.a` renderer, or any ancestor in the React component tree up to `MessageContent`. The `aria-hidden="true"` attributes in `MessageBubble` are confined to SVG icons (lines 14, 33), the streaming cursor span (lines 242, 262), the directional arrow (line 401), and the warning icon (line 410). None are ancestors of rendered markdown links.
- The `MessageContent` div for assistant messages (lines 249–266) uses `aria-live` and `aria-atomic` but not `aria-hidden`.
- `sr-only` in Tailwind v3 uses `position: absolute; width: 1px; height: 1px; padding: 0; margin: -1px; overflow: hidden; clip: rect(0, 0, 0, 0); white-space: nowrap; border-width: 0` — this hides the element visually while keeping it in the accessibility tree. No `display: none` or `visibility: hidden` is applied.

**rehype-sanitize interaction**: Confirmed that `rehypeSanitize` (which uses `hast-util-sanitize`'s `defaultSchema`) operates on the HAST tree before React rendering. The `components` prop provides JSX-level renderers that execute after the sanitize pass. The `<span className="sr-only">` is produced by the React renderer, not by the markdown input — it cannot be stripped by `rehypeSanitize`. The `sr-only` class survives to the DOM. (Incidental note: even if the span were in markdown HTML, `span` is in `defaultSchema.tagNames`, though `className` is not in the default span attribute allowlist — the class would be stripped but the element would remain. This path is moot since the span originates from the component renderer, not markdown input.)

**WCAG 2.4.4** (Link Purpose in Context): Screen reader users will hear the link text followed by "(opens in new tab)" for all external links. Pass.

### 3. External link detection (`isExternal` condition) — PASS

**Finding**: The condition at line 154 is:

```ts
const isExternal = href?.startsWith('http') || href?.startsWith('//');
```

This correctly identifies:
- `http://...` and `https://...` (startsWith `'http'`) — all absolute HTTP/HTTPS URLs
- `//example.com` (protocol-relative URLs) — correctly treated as external

Internal links that would not be caught:
- `mailto:`, `tel:`, `ftp:` — these do not receive `target="_blank"` (correct: the spread is conditional on `isExternal`), and do not receive the sr-only announcement. This is correct behavior: these are not browser-tab-opening links.
- Relative paths (`./`, `../`, `/path`) — correctly excluded. These are anchor or in-app navigation links, not new-tab links.
- Fragment-only links (`#section`) — correctly excluded.

The `target="_blank"` and `rel="noopener noreferrer"` are gated on the same `isExternal` check (line 158–161), so the sr-only span is injected if and only if `target="_blank"` is present. The condition is consistent and correct.

**No false negatives**: All genuinely external links (http, https, protocol-relative) receive both `target="_blank"` and the sr-only announcement.  
**No false positives**: Internal, fragment, and non-http protocol links do not receive either.

Pass.

## Advisory Finding (Pre-existing — Not a PR Blocker)

**AV1 — No persistent `<h1>` in `<main>` during conversation view**

When `OnboardingEmptyState` is not rendered (i.e. at least one provider is configured), there is no `<h1>` or `<h2>` in the `<main>` landmark. The highest heading in `<main>` is `<h3>` (from model-generated markdown). While WCAG 2.1 AA does not strictly require a `<h1>` per page under all circumstances, best practice (and WCAG 2.4.6 advisory) recommends a visible or visually-hidden heading at the start of the main content landmark to orient screen reader users.

This is pre-existing and not introduced by this PR. The PR's heading downshift actually improves the situation (previously `<h1>` appeared in message content, which would confuse AT users expecting the page-level `<h1>`). Recommended fix: add a visually-hidden `<h1>` ("Roundtable" or "Conversation") as the first child of `<main>` in `AppLayout.tsx`. File as a new issue for Aria.

**Action**: Open a new issue — do not block merge.
